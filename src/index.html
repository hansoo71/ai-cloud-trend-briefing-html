왜 지금 중요한가

지난 24시간의 발표를 하나의 흐름으로 보면 시장의 무게중심이 분명합니다. Google은 Gemini를 검색, Workspace, 개발도구, 쇼핑까지 확장하며 AI를 제품 안의 기능이 아니라 사용자의 업무 표면 전체를 움직이는 에이전트 레이어로 만들고 있습니다. OpenAI의 Ramp 사례는 Codex가 “개발자의 보조 챗봇”을 넘어 코드 리뷰와 온콜 도구 개발 같은 운영 워크플로우에 들어갔다는 점을 보여줍니다. Databricks는 에이전트가 수천 개로 늘어날 때 누가 어떤 데이터와 도구를 호출했는지 답하지 못하는 조직의 리스크를 지적했고, AWS와 Harness는 권한·평가·CI 비용·공급망 보안이라는 운영 이슈를 전면에 올렸습니다.

즉 고객이 지금 고민해야 할 질문은 “어떤 LLM이 가장 좋은가?”가 아닙니다. “우리 회사의 AI가 어떤 데이터에 접근하고, 어떤 도구를 호출하며, 그 결과를 어떻게 검증하고, 비용과 감사 로그를 누가 책임지는가?”입니다. 이 질문에 답하지 못하면 PoC는 화려해도 운영 전환 단계에서 멈춥니다.

핵심 변화 3가지

첫째, 컨텍스트 엔지니어링이 데이터 플랫폼 설계로 진화하고 있습니다. Google Cloud Data Agent Kit은 Claude Code, Codex, Gemini CLI 같은 개발 환경과 BigQuery, AlloyDB, Cloud Storage를 연결합니다. 이는 문서 몇 개를 프롬프트에 붙이는 수준을 넘어, 에이전트가 업무 맥락을 안정적으로 가져오기 위한 권한·메타데이터·도구 연결 계층이 필요하다는 뜻입니다. 기업 고객에게는 MCP 서버, 데이터 카탈로그, 권한 정책, 로그 수집이 모두 컨텍스트 엔지니어링의 구성요소가 됩니다.

둘째, AI Agent 거버넌스가 보안·데이터 조직의 공동 과제가 됐습니다. Databricks의 Unity Catalog 접근과 AWS Security Hub의 미사용 권한 탐지는 같은 메시지를 말합니다. 에이전트가 도구를 호출하고 데이터에 접근할수록 과잉권한은 곧 비즈니스 리스크가 됩니다. 특히 금융·공공·제약 고객은 “AI가 무엇을 할 수 있는가”보다 “AI가 무엇을 했는지 증명할 수 있는가”를 먼저 요구하게 됩니다.

셋째, AI 품질 기준이 데모 품질에서 운영 품질로 올라가고 있습니다. AWS Strands Evals의 멀티모달 평가기는 이미지 기반 답변이 실제 이미지에 근거했는지 자동 평가합니다. Harness의 CI 비용·공급망 보안 글은 AI Coding이 개발 속도를 높이는 동시에 빌드·테스트·패키지 변경·토큰 노출을 증가시킬 수 있음을 상기시킵니다. 앞으로의 제안은 모델 성능표보다 평가 자동화, 릴리즈 게이트, 비용 관측, 공급망 통제까지 포함해야 설득력이 생깁니다.

Presales 관점의 핵심 메시지
AI Agent 도입 제안은 “모델 사용권 판매”가 아니라 “운영모델 전환”으로 포지셔닝해야 합니다. 고객의 데이터 위치, 권한 구조, 개발 파이프라인, 품질 평가 방식, 비용 책임 구조를 함께 진단하는 워크숍이 가장 자연스러운 진입점입니다.

실무자에게 미치는 영향

메가존/AIR 관점에서는 세 가지 패키지로 대화가 가능합니다. 첫째, Agent Readiness Assessment입니다. 고객의 주요 업무 프로세스, 데이터 소스, 권한 체계, 감사 로그, 비용 관리 상태를 점검해 PoC 가능 영역과 운영 전환 리스크를 분리합니다. 둘째, Context & Governance Architecture입니다. Gemini Enterprise/Vertex AI, BigQuery, Databricks Unity Catalog, AWS Security Hub, MCP/AI Gateway를 고객 환경에 맞게 조합해 에이전트가 안전하게 일하는 경계를 설계합니다. 셋째, AI Engineering Operating Model입니다. Codex류 개발 도구 도입과 함께 코드 리뷰 기준, CI/CD 비용, 공급망 보안, 평가 자동화를 묶어 개발 조직의 생산성뿐 아니라 통제력을 높입니다.

고객 미팅에서는 다음 질문이 유효합니다. “AI가 접근해야 하는 핵심 데이터는 어디에 있으며, 권한은 사람 기준입니까 애플리케이션 기준입니까?”, “에이전트가 호출한 도구와 데이터 접근 로그를 한 화면에서 볼 수 있습니까?”, “AI Coding 도입 이후 PR 수, 빌드 수, 테스트 실패 재시도 비용이 늘었습니까?”, “멀티모달/OCR/차트 분석 결과를 사람이 샘플링 검수합니까, 아니면 자동 평가 게이트가 있습니까?” 이 질문들은 단순 관심 확인이 아니라 컨설팅 과제의 범위를 자연스럽게 정의합니다.

결론

AI Agent의 다음 단계는 더 큰 데모가 아니라 더 안전한 운영입니다. 컨텍스트, 권한, 평가, 비용, 감사 로그가 하나의 설계 안에 들어오지 않으면 에이전트는 빠르게 통제 불가능한 자동화가 됩니다. 반대로 이 다섯 가지를 먼저 설계하면 고객은 단순 챗봇을 넘어 실제 업무를 수행하는 AI 운영체계를 만들 수 있습니다. 지금의 시장 신호는 명확합니다. AI 도입의 승부처는 모델 선택이 아니라, 모델이 일할 수 있는 엔터프라이즈 운영 구조를 누가 먼저 설계하느냐입니다.
