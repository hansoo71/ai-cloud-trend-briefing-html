  - 공개 글은 AI 에이전트가 텔레메트리, 변경 이력, 의존성, 런북을 모아 가설을 제시하고, 권한 경계와 confidence routing 아래에서만 조치를 수행해야 한다고 강조한다.
  - 이는 CBA 사례와 함께 “운영 사고 대응”이 에이전트 도입의 현실적인 첫 타깃임을 보여준다.
- 실무 관점 코멘트:
  - 고객 제안은 AIOps를 대체하는 거창한 플랫폼보다 기존 Observability·ITSM·CI/CD와 연결되는 incident co-pilot로 시작하는 편이 설득력 있다.
  - 성공 KPI는 MTTA/MTTR, 알림 중복 제거율, evidence pack 생성시간, 승인된 자동조치율로 잡을 수 있다.


## 3. 자체 작성 블로그

## 제목
AI 에이전트의 다음 경쟁력은 모델 성능이 아니라 운영모델이다

## 왜 지금 중요한가
지난 24시간의 신호는 꽤 일관적이다. CBA는 새벽 온콜 엔지니어의 cold start를 줄이기 위해 AWS 기반 DevOps 에이전트를 실험하고 있고, Rio Tinto는 30년 된 제조 실행 시스템의 지식과 의존성을 Bedrock·AgentCore·SageMaker 기반 어시스턴트로 operationalize하고 있다. 동시에 InfoWorld는 MCP를 컨텍스트 엔지니어링의 실행 계층으로 설명했고, Google Cloud와 Microsoft 관련 보안 글들은 AI 시스템 자체가 새로운 공격·검증 표면이 되고 있음을 보여준다.

이 흐름의 핵심은 “더 똑똑한 모델을 붙이면 된다”가 아니다. 기업 고객이 실제로 요구하는 것은 모델보다 운영모델이다. 어떤 데이터와 도구를 연결할지, 에이전트가 어떤 권한으로 실행할지, 사고가 났을 때 누가 승인하고 감사할지, 비용과 품질을 어떤 KPI로 관리할지가 구매 의사결정의 중심으로 이동하고 있다.

## 핵심 변화 3가지
첫째, 컨텍스트는 프롬프트가 아니라 아키텍처가 되었다. MCP, Knowledge Bases, 코드·문서·운영 로그 연결은 모델에 넣는 문장 몇 줄의 문제가 아니다. 데이터 신선도, 권한 필터링, retrieval 품질, 도구 호출 실패 처리, 감사 로그까지 포함한 control plane 설계가 필요하다.

둘째, 에이전트의 첫 생산 유스케이스는 완전 자동화가 아니라 전문가 보조다. CBA의 온콜 사례와 Augment Code의 AI SRE 논의는 에이전트가 사람을 대체하기보다 증거 수집, 영향 범위 분석, 초기 가설 생성, 런북 검색을 통해 “초기 대응 시간을 줄이는” 역할부터 시작한다는 점을 보여준다. Rio Tinto 사례도 마찬가지다. 레거시 시스템을 교체하지 않고, 흩어진 지식과 의존성을 엔지니어가 사용할 수 있는 형태로 바꾸는 것이 첫 가치다.

셋째, 보안은 사후 검사가 아니라 에이전트 운영의 내장 조건이 되었다. Google Cloud의 ViewState 취약점 분석, 중국어권 PhaaS 분석, Rescana의 CI/CD 공급망 공격, Microsoft MDASH 보도는 AI 도입이 보안팀의 통제 밖에서 확산될 때 어떤 리스크가 생기는지 보여준다. 에이전트가 클라우드 API, GitHub Actions, 업무 SaaS, 지식베이스를 호출한다면 기존 애플리케이션보다 더 엄격한 신원·권한·감사 체계가 필요하다.

## 엔터프라이즈 도입 시 리스크
가장 큰 리스크는 에이전트가 “업무를 잘 아는 듯 보이지만 실제 권한과 책임 경계가 불명확한 상태”로 운영되는 것이다. 도구 호출 권한이 넓고 로그가 부족하면 사고 조사 시 왜 어떤 조치가 실행되었는지 설명하기 어렵다. RAG 결과가 오래되었거나 권한 필터링이 부정확하면 내부 문서 유출 또는 잘못된 의사결정으로 이어질 수 있다. 또한 모델 제공자에 종속된 아키텍처는 비용, latency, 정책 변경에 취약하다.

따라서 제안 단계에서는 모델 데모보다 운영 체크리스트를 먼저 보여주는 편이 효과적이다. 데이터 경계, MCP 서버 inventory, tool allowlist, secret handling, human approval point, incident rollback, 평가 데이터셋, 비용 모니터링을 한 장의 reference architecture로 정리하면 고객은 “우리 조직에 적용 가능한지”를 더 빨리 판단할 수 있다.

## 실무자에게 미치는 영향
Presales와 컨설팅팀은 AI PoC를 기능 중심으로 팔기보다 운영 패키지로 재구성해야 한다. 예를 들어 금융권에는 온콜/장애 triage agent, 제조에는 레거시 시스템 지식 어시스턴트, 유통에는 피싱·브랜드 악용 모니터링과 고객지원 agent, 공공에는 데이터 주권과 감사 중심의 private AI agent platform이 첫-fit이 될 수 있다.

개발·플랫폼팀은 AI Gateway, model-agnostic routing, 관측성, prompt/version 관리, 정책 기반 도구 호출을 준비해야 한다. 보안팀은 AI red-team과 MCP/tool 권한 리뷰를 정기 점검 항목으로 넣어야 한다. 운영팀은 에이전트가 만든 evidence pack을 사고 대응 프로세스에 편입하고, 자동조치 범위를 낮은 위험부터 단계적으로 넓혀야 한다.

## 고객 미팅 질문
1. 가장 많은 시간이 낭비되는 장애 대응 단계는 알림 분류, 로그 수집, 원인 가설, 승인, 복구 중 어디인가?
2. 레거시 시스템 중 문서가 흩어져 있고 변경 영향도를 알기 어려운 영역은 무엇인가?
3. 에이전트가 호출할 수 있는 도구와 데이터 소스 목록을 누가 승인하고, 호출 로그는 어디에 남길 것인가?
4. 모델 제공자 변경, 장애, 비용 급등 시 대체 모델 또는 라우팅 정책이 있는가?
5. AI 생성 코드와 CI/CD workflow가 비밀정보·토큰·외부 action을 어떻게 다루는지 점검하고 있는가?

## 결론
오늘의 브리핑에서 가장 중요한 메시지는 명확하다. AI 에이전트 경쟁은 모델 성능 경쟁에서 운영모델 경쟁으로 이동하고 있다. 컨텍스트를 잘 연결하는 기업, 권한과 감사를 명확히 하는 기업, 운영 KPI와 비용을 투명하게 관리하는 기업이 실제 production AI의 승자가 된다. 따라서 지금 필요한 제안은 “AI를 붙여드립니다”가 아니라 “안전하고 효율적인 AI 에이전트 운영체계를 설계해드립니다”여야 한다.
